# 資料庫（Database）：結構化資料的基石

## ### 1. 核心概念與定義

#### 什麼是資料庫？

**定義/核心觀念：**
資料庫（Database）是指經過組織、儲存和管理的資料集合。它的核心目的是為了有效地儲存大量資料，並且方便使用者進行快速的查詢、新增、修改與刪除。這些資料通常以結構化的形式存在，以便於電腦系統進行處理。

**例子或推導：**
想像一下，你家裡的電話簿就是一個非常簡單的「資料庫」。它儲存了朋友的姓名、電話號碼、地址等資訊。如果你要找某位朋友的電話，你可能會按照姓名筆畫或字母順序去查找，這就類似於資料庫的查詢操作。

一個更具體的電腦化例子是學校的學生管理系統。它可能包含：
- 學生資料表：學號、姓名、出生日期、班級。
- 課程資料表：課程代碼、課程名稱、學分。
- 成績資料表：學號、課程代碼、分數。

這些表格之間透過學號、課程代碼等欄位互相連接，形成一個有組織的資料集合。

**與相鄰概念的關聯：**
- **資料 (Data)：** 原始、未經處理的事實、數字或文字。例如，「John Doe」、「12345」都是資料。
- **資訊 (Information)：** 經過處理、組織和解釋的資料，使其具有意義和上下文。例如，「John Doe 的電話號碼是 123-4567」就是資訊。
- 資料庫是儲存和管理「資料」的系統，目的是為了讓這些資料能被有效地轉換為「資訊」並加以利用。

-----

#### 資料庫管理系統 (DBMS)

**定義/核心觀念：**
資料庫管理系統（Database Management System, DBMS）是一種軟體，用於建立、管理和操作資料庫。它充當使用者、應用程式與實際資料庫之間的介面，負責資料的儲存、安全、完整性、備份與恢復等功能。沒有 DBMS，我們很難直接有效地與龐大的資料庫互動。

**例子或推導：**
如果你家電話簿是資料庫，那麼你本人就是「DBMS」，因為你負責翻找、新增、劃掉舊號碼等操作。

在電腦世界中，常見的 DBMS 有：
- **關聯式資料庫管理系統 (RDBMS)：** MySQL, PostgreSQL, Oracle Database, SQL Server。
- **NoSQL 資料庫管理系統：** MongoDB (文件型), Redis (鍵值型), Cassandra (寬列型), Neo4j (圖形型)。

當一個應用程式需要讀取學生的成績時，它會透過 DBMS 發送一個查詢指令（例如 SQL 語句），DBMS 收到指令後，會去底層的資料庫檔案中找出對應的資料，然後回傳給應用程式。

**與相鄰概念的關聯：**
- **資料庫與 DBMS 的共生關係：** 資料庫是資料的「容器」，而 DBMS 則是操作和管理這個容器的「工具」。兩者密不可分，通常我們口語說的「使用資料庫」其實是指「使用 DBMS 來操作資料庫」。
- **應用程式：** 應用程式（例如網站、手機 App）透過 DBMS 提供定義好的介面（API）來存取資料庫。

-----

#### 資料庫的三層架構（Three-Schema Architecture）

**定義/核心觀念：**
資料庫三層架構是一種標準的資料庫設計模型，旨在實現資料獨立性。它將資料庫的視圖分為三個層次：
1.  **外部層 (External Level / View Level)：** 使用者或應用程式所看到的資料視圖。每個使用者可能只看到資料庫中自己關心的那部分資料，並且可以以不同的方式呈現。
2.  **概念層 (Conceptual Level)：** 整個資料庫的邏輯結構視圖。它描述了資料庫中儲存了哪些實體（Entity）、這些實體的屬性（Attribute）以及實體之間的關係（Relationship），但沒有涉及實際的儲存細節。
3.  **內部層 (Internal Level / Physical Level)：** 資料庫的物理儲存結構視圖。它描述了資料是如何實際儲存到儲存介質上（例如硬碟），包括檔案結構、索引、記錄的物理位置等。

**例子或推導：**
想像一間大型圖書館的運作：
-   **外部層：** 讀者透過圖書館網站或查詢機搜尋書籍。不同讀者可能看到不同的查詢介面，兒童讀者可能看到卡通界面，研究者則看到學術搜索界面。但他們都只關心「是否有這本書」、「書在哪裡」等問題。
-   **概念層：** 圖書館管理員會有一份完整的清單，列出所有書籍、作者、出版社、分類、出版年份等資訊，以及這些資訊之間的關聯（一本書屬於一位作者，一位作者可寫多本書）。這份清單完整描述了圖書館所有書籍的邏輯結構。
-   **內部層：** 圖書館管理員知道哪些書放在哪些書架上，哪些書是電子書儲存在伺服器上，書籍的條碼是如何編碼的，以及如何建立索引以加快查找速度。這是實際的儲存細節。

**優點：資料獨立性**
三層架構最主要的優點是實現了**資料獨立性 (Data Independence)**：
-   **邏輯資料獨立性：** 改變概念層（例如，在學生表中增加一個新的「國籍」欄位），不需要修改外部層的應用程式。
-   **物理資料獨立性：** 改變內部層（例如，將資料庫從 HDD 移到 SSD，或改變檔案儲存方式），不需要修改概念層或外部層的應用程式。

這使得資料庫系統更具彈性，易於維護和演進。

## ### 2. 典型例子與資料模型

#### 資料模型 (Data Models)

**定義/核心觀念：**
資料模型是資料庫中資料組織和表示的抽象框架。它定義了資料如何被描述、儲存和處理，以及資料之間的關係。選擇合適的資料模型是設計高效能資料庫的關鍵。

**與相鄰概念的關聯：**
資料模型是 DBMS 的核心，不同的 DBMS 通常基於不同的資料模型。例如，MySQL 是基於關聯式資料模型的 RDBMS，而 MongoDB 則是基於文件資料模型的 NoSQL DBMS。

-----

#### 關聯式資料模型 (Relational Data Model)

**定義/核心觀念：**
關聯式資料模型是目前最廣泛使用的資料模型。它將資料以二維表格（Table）的形式組織起來。每個表格由多個「列」（Row 或 Tuple）和多個「欄」（Column 或 Attribute）組成。
-   **表格 (Table)：** 用來儲存特定類型實體（如「學生」或「課程」）的資料。
-   **列 (Row/Tuple)：** 表格中的一筆記錄，代表一個實體實例（如一個特定的學生）。
-   **欄 (Column/Attribute)：** 表格中的一個屬性，代表實體的一個特性（如學生的「姓名」或「學號」）。
-   **主鍵 (Primary Key)：** 表格中一個或一組欄位，其值必須唯一且不能為空，用於唯一識別每一列。
-   **外鍵 (Foreign Key)：** 表格中一個或一組欄位，其值指向另一個表格的主鍵。外鍵用於建立兩個表格之間的關聯。

**例子或推導：**
我們以一個學校的簡單學生與課程資料庫為例：

**表格一：學生 (Students)**

| 學號 (StudentID) [主鍵] | 姓名 (Name) | 性別 (Gender) | 出生日期 (DOB) |
| :--------------------: | :---------: | :-----------: | :------------: |
|         S001           |    王大明   |      男       |   1998-05-12   |
|         S002           |    陳小華   |      女       |   1999-03-20   |
|         S003           |    林美玲   |      女       |   1998-11-01   |

**表格二：課程 (Courses)**

| 課程代碼 (CourseID) [主鍵] | 課程名稱 (CourseName) | 學分 (Credits) |
| :----------------------: | :-------------------: | :------------: |
|           C101           |       資料庫導論      |       3        |
|           C102           |       程式設計       |       4        |
|           C103           |        線性代數       |       3        |

**表格三：選課 (Enrollments)**

| 學號 (StudentID) [外鍵] | 課程代碼 (CourseID) [外鍵] | 成績 (Grade) |
| :--------------------: | :-----------------------: | :----------: |
|         S001           |           C101           |      A       |
|         S001           |           C102           |      B+      |
|         S002           |           C101           |      B       |
|         S003           |           C103           |      A-      |

-   `StudentID` 在 `Students` 表格中是主鍵。
-   `CourseID` 在 `Courses` 表格中是主鍵。
-   在 `Enrollments` 表格中，`StudentID` 和 `CourseID` 都是外鍵，它們分別指向 `Students` 表格的 `StudentID` 和 `Courses` 表格的 `CourseID`。
-   `Enrollments` 表格的組合主鍵可以是 (`StudentID`, `CourseID`)，因為一個學生不能選同一門課兩次。

這種透過主鍵和外鍵建立表格之間關聯的方式，是關聯式資料模型的核心。

-----

#### 其他資料模型簡介

除了關聯式資料模型，還有許多其他資料模型，特別是在 NoSQL 運動興起後：

-   **文件型資料模型 (Document-oriented)：**
    -   **核心觀念：** 資料以半結構化的「文件」形式儲存，通常是 JSON 或 BSON 格式。文件可以包含嵌套結構，且不同文件可以有不同的欄位（彈性 schema）。
    -   **適用場景：** 內容管理、目錄、使用者設定檔。
    -   **代表 DBMS：** MongoDB, Couchbase。

-   **鍵值型資料模型 (Key-Value)：**
    -   **核心觀念：** 最簡單的資料模型，每個資料項都是一個鍵值對。鍵是唯一的，用於快速檢索值。值可以是任何類型（字串、數字、物件）。
    -   **適用場景：** 緩存、會話管理、即時數據收集。
    -   **代表 DBMS：** Redis, Amazon DynamoDB。

-   **圖形資料模型 (Graph)：**
    -   **核心觀念：** 資料以節點（Nodes）和邊（Edges）的形式儲存，節點代表實體，邊代表實體之間的關係。這種模型能高效處理複雜的關聯查詢。
    -   **適用場景：** 社交網絡、推薦系統、欺詐檢測。
    -   **代表 DBMS：** Neo4j, Amazon Neptune。

-   **寬列型資料模型 (Wide-Column)：**
    -   **核心觀念：** 類似於關聯式表格，但列可以是動態的，且每行不必具有相同的列。為分散式儲存和極高吞吐量而設計。
    -   **適用場景：** 大數據分析、時間序列數據、日誌記錄。
    -   **代表 DBMS：** Apache Cassandra, Apache HBase。

## ### 3. 與相鄰概念的關聯

#### 資料庫與應用程式

資料庫是絕大多數現代應用程式的骨幹。應用程式（無論是網頁應用、行動應用還是桌面應用）通常需要存儲和檢索數據。
-   **互動方式：** 應用程式透過程式語言提供的資料庫驅動程式（如 Python 的 `psycopg2` for PostgreSQL, Java 的 JDBC）連接到 DBMS。
-   **操作介面：** 程式碼中會嵌入 SQL 語句（或使用 ORM 工具如 SQLAlchemy、Hibernate 將物件操作轉換為 SQL），透過 DBMS 執行資料庫操作。
-   **範例：** 一個電子商務網站需要資料庫來儲存商品資訊、使用者帳戶、訂單記錄。當使用者瀏覽商品、下訂單時，網站應用程式會與資料庫互動來讀取或寫入數據。

-----

#### 資料庫與資料倉儲 (Data Warehouse)

儘管兩者都儲存資料，但它們的設計目標和使用場景截然不同。

| 特性     | 資料庫 (OLTP)                       | 資料倉儲 (OLAP)                     |
| :------- | :---------------------------------- | :---------------------------------- |
| **目的** | 支援日常交易處理（Transactional Processing） | 支援決策分析（Analytical Processing） |
| **資料** | 即時、最新資料，常態化（Normalized） | 歷史、整合資料，反常態化（Denormalized） |
| **操作** | 頻繁的讀取、寫入、更新、刪除        | 大量的讀取、複雜查詢（彙總、報表）  |
| **使用者** | 營運人員、最終使用者                 | 商業分析師、資料科學家              |
| **性能** | 強調事務處理速度、資料完整性        | 強調查詢響應速度、分析效率          |

**關聯：** 資料倉儲的資料通常來自多個操作型資料庫（transactional databases）和其他資料來源，經過提取（Extract）、轉換（Transform）、載入（Load）的 ETL 過程後，統一儲存。

-----

#### 資料庫與雲端計算 (Cloud Computing)

雲端計算極大地改變了資料庫的部署和管理方式。
-   **DBaaS (Database as a Service)：** 雲服務提供商提供資料庫服務，使用者無需管理底層基礎設施（伺服器、儲存、網路），只需專注於資料庫的資料和應用。
    -   **範例：** Amazon RDS (for MySQL, PostgreSQL, SQL Server etc.), Google Cloud SQL, Azure SQL Database, MongoDB Atlas。
-   **優勢：** 彈性擴展、高可用性、降低營運成本、自動備份與更新。
-   **關聯：** 雲端資料庫使得資料庫的部署和維護變得更加便捷和成本效益高，尤其適合初創企業和需要快速擴展的應用。

## ### 4. 進階內容：SQL 與正規化

#### 資料庫語言 SQL (Structured Query Language)

SQL 是用於管理關聯式資料庫的標準語言。它允許使用者定義資料結構、操作資料、控制資料存取。

**核心觀念：** SQL 分為幾個子語言：

1.  **資料定義語言 (DDL - Data Definition Language)：** 用於定義、修改和刪除資料庫物件（如表、視圖、索引）。
    -   `CREATE TABLE`：建立新表。
    -   `ALTER TABLE`：修改表結構。
    -   `DROP TABLE`：刪除表。
    -   `CREATE INDEX`：建立索引。

2.  **資料操作語言 (DML - Data Manipulation Language)：** 用於查詢、新增、修改和刪除資料。
    -   `SELECT`：從資料庫中檢索資料。
    -   `INSERT`：向表中插入新資料。
    -   `UPDATE`：修改表中現有資料。
    -   `DELETE`：從表中刪除資料。

3.  **資料控制語言 (DCL - Data Control Language)：** 用於管理使用者權限和資料庫安全性。
    -   `GRANT`：授予使用者權限。
    -   `REVOKE`：撤銷使用者權限。

4.  **事務控制語言 (TCL - Transaction Control Language)：** 用於管理事務（一系列操作的原子單元）。
    -   `COMMIT`：確認事務並將更改永久保存。
    -   `ROLLBACK`：撤銷事務並回滾到上次 `COMMIT` 的狀態。

**例子：**
```sql
-- DDL: 建立一個新的客戶表
CREATE TABLE Customers (
    CustomerID INT PRIMARY KEY,
    FirstName VARCHAR(50),
    LastName VARCHAR(50),
    Email VARCHAR(100) UNIQUE
);

-- DML: 插入一筆新客戶資料
INSERT INTO Customers (CustomerID, FirstName, LastName, Email)
VALUES (1, '張', '三', 'zhangsan@example.com');

-- DML: 查詢所有客戶的姓名和電子郵件
SELECT FirstName, LastName, Email FROM Customers;

-- DML: 更新客戶的電子郵件
UPDATE Customers SET Email = 'zhangsan_new@example.com' WHERE CustomerID = 1;

-- DCL: 授予用戶 'analyst' 對 Customers 表的 SELECT 權限
GRANT SELECT ON Customers TO analyst;
```

#### 關聯式資料庫的正規化 (Normalization)

**定義/核心觀念：**
正規化是一種資料庫設計技術，用於組織關聯式資料庫的表格，以減少資料冗餘（Data Redundancy）和提高資料完整性（Data Integrity）。其目標是分解大型表格為較小的、相關聯的表格，並定義它們之間的關係。

正規化過程遵循一系列「正規形式」（Normal Forms），最常見的是：

-   **第一正規形式 (1NF)：**
    -   表格的每個欄位都包含原子值（不可再分）。
    -   每個欄位只包含單一值，不允許重複群組。
    -   例子：一個儲存多個電話號碼的欄位不符合 1NF。應該為每個電話號碼創建單獨的列或單獨的表。

-   **第二正規形式 (2NF)：**
    -   符合 1NF。
    -   所有非主鍵欄位都必須完全依賴於整個主鍵。如果主鍵是複合鍵，則不能存在非主鍵欄位只依賴於主鍵的一部分。
    -   例子：如果複合主鍵是 (訂單ID, 產品ID)，而產品名稱只依賴於產品ID（而非訂單ID），則此表格不符合 2NF。應將產品資訊拆分到一個單獨的產品表。

-   **第三正規形式 (3NF)：**
    -   符合 2NF。
    -   所有非主鍵欄位都不能有遞移依賴（Transitive Dependency）。即，非主鍵欄位不能依賴於另一個非主鍵欄位。
    -   例子：如果員工表格包含「部門名稱」和「部門地址」，而「部門地址」只依賴於「部門名稱」（一個非主鍵），則此表格不符合 3NF。應將部門資訊拆分到一個單獨的部門表。

**為什麼需要正規化？**
1.  **減少資料冗餘：** 避免相同資料在多個地方重複儲存，節省儲存空間。
2.  **避免更新異常：**
    -   **插入異常：** 無法在不插入其他資訊的情況下插入新的實體。
    -   **刪除異常：** 刪除某些資訊時意外地丟失了其他重要資訊。
    -   **更新異常：** 更新同一數據時需要修改多個地方，容易導致資料不一致。
3.  **提高資料完整性：** 確保資料的一致性和準確性。

**反正規化 (Denormalization)：**
在某些情況下，為了提高查詢效能，可以有意地引入一些冗餘，這個過程稱為反正規化。這是一種權衡，用儲存空間和一些更新風險來換取讀取速度，通常用於資料倉儲或需要快速產生複雜報表的場景。

## ### 5. 常見錯誤與澄清

#### 混淆資料庫與 DBMS

-   **錯誤：** 認為「資料庫」和「DBMS」是同一個東西，可互換使用。
-   **澄清：**
    -   **資料庫 (Database)：** 是儲存資料的集合，就像一個裝滿書的圖書館。
    -   **DBMS (Database Management System)：** 是一個軟體，用於管理、操作這個資料集合，就像圖書館的管理系統或圖書館員。
    -   你可以沒有 DBMS 但有一個資料庫（例如，一份手寫的電話簿），但電腦化的資料庫幾乎總是需要 DBMS 來管理。

#### 誤解主鍵與唯一鍵的差異

-   **錯誤：** 認為主鍵（Primary Key）和唯一鍵（Unique Key）是完全一樣的，或者唯一鍵可以為空值。
-   **澄清：**
    -   **主鍵 (Primary Key)：**
        -   **唯一性：** 值必須是唯一的。
        -   **非空性 (NOT NULL)：** 不能包含 `NULL` 值。
        -   **唯一性目的：** 唯一識別表格中的每一列。一個表格只能有一個主鍵。
    -   **唯一鍵 (Unique Key)：**
        -   **唯一性：** 值必須是唯一的。
        -   **可空性：** 可以包含一個或多個 `NULL` 值（根據不同的 DBMS 實現，通常只允許一個 `NULL` 或多個 `NULL` 均視為唯一）。
        -   **唯一性目的：** 確保特定欄位的值是唯一的，但該欄位不一定是行的主要識別符。一個表格可以有多個唯一鍵。
    -   **例子：** 學生的 `學號` 是主鍵（唯一且非空）。學生的 `身分證字號` 也應該是唯一的，可以設為唯一鍵（雖然在某些國家這可能與主鍵一樣重要）。學生的 `電子郵件` 也可能設為唯一鍵，但可以暫時為空（如果學生還沒有提供）。

#### 過度正規化或不足正規化

-   **錯誤：** 盲目追求最高正規形式，或完全不做正規化。
-   **澄清：**
    -   **過度正規化：** 將表格拆分得過於細碎，導致執行簡單查詢時需要頻繁的 `JOIN` 操作，反而降低查詢效能。在某些情況下，微量的資料冗餘（即反正規化）可以提高效能。
    -   **不足正規化：** 表格中存在大量冗餘和資料依賴問題，導致資料更新異常、儲存空間浪費和資料一致性問題。
    -   **最佳實踐：** 通常情況下，資料庫設計達到第三正規形式 (3NF) 即可滿足大多數應用需求，並在必要時進行適度的反正規化以優化讀取效能。這是一個需要在設計時權衡取捨的問題。

## ### 6. 小練習（附詳解）

#### 小練習一：關聯式表格設計

**情境：**
你被要求設計一個簡單的圖書館資料庫，用於管理圖書和作者資訊。
需求如下：
-   每本書有唯一的國際標準書號（ISBN）、書名、出版年份。
-   每位作者有唯一的作者ID、姓名、出生年份。
-   一本書可以有多位作者，一位作者也可以寫多本書。

**目標：**
請設計出至少三個關聯式表格，並標註主鍵和外鍵，以滿足上述需求。

**步驟：**
1.  **識別實體：** 從情境描述中找出主要的「事物」。
2.  **識別屬性：** 為每個實體找出其相關的「特性」。
3.  **定義主鍵：** 為每個表格選擇一個或一組能唯一識別每條記錄的欄位作為主鍵。
4.  **建立關係：** 考量實體間的關係（例如：一對多、多對多），並引入外鍵或關聯表來表示這些關係。

**詳解：**

1.  **識別實體與屬性：**
    -   **圖書 (Books)：** ISBN (唯一的), 書名 (Title), 出版年份 (PublicationYear)。
    -   **作者 (Authors)：** 作者ID (AuthorID, 唯一的), 姓名 (Name), 出生年份 (BirthYear)。
    -   **關係 (多對多)：** 一本書有多個作者，一個作者寫多本書。這需要一個中間的「關聯表」來處理多對多關係。

2.  **設計表格：**

    **表格一：書籍 (Books)**

    | 欄位名稱           | 資料類型      | 備註         |
    | :----------------- | :------------ | :----------- |
    | `ISBN`             | VARCHAR(13)   | **主鍵**     |
    | `Title`            | VARCHAR(255)  | 書名         |
    | `PublicationYear`  | INT           | 出版年份     |

    **表格二：作者 (Authors)**

    | 欄位名稱         | 資料類型     | 備註       |
    | :--------------- | :----------- | :--------- |
    | `AuthorID`       | INT          | **主鍵**   |
    | `Name`           | VARCHAR(100) | 作者姓名   |
    | `BirthYear`      | INT          | 出生年份   |

    **表格三：書籍作者關聯 (BookAuthors)**
    這個表格用於處理書籍和作者之間的多對多關係。它的主鍵由兩個外鍵組成。

    | 欄位名稱         | 資料類型      | 備註                                   |
    | :--------------- | :------------ | :------------------------------------- |
    | `ISBN`           | VARCHAR(13)   | **外鍵** 參考 `Books.ISBN`，且是複合主鍵的一部分 |
    | `AuthorID`       | INT           | **外鍵** 參考 `Authors.AuthorID`，且是複合主鍵的一部分 |
    | `PRIMARY KEY`    | (`ISBN`, `AuthorID`) | **複合主鍵**                          |

    **關聯說明：**
    -   `BookAuthors.ISBN` 是外鍵，指向 `Books.ISBN`。
    -   `BookAuthors.AuthorID` 是外鍵，指向 `Authors.AuthorID`。
    -   透過 `BookAuthors` 表，我們可以知道哪本書是由哪些作者寫的，以及哪些作者寫了哪些書。

#### 小練習二：SQL 基本查詢

**情境：**
延續小練習一的圖書館資料庫設計，假設我們已經按照設計建立了表格並插入了以下數據：

**Books 表：**

| ISBN         | Title        | PublicationYear |
| :----------- | :----------- | :-------------- |
| 978-0321765723 | The C++ Programming Language | 2013            |
| 978-0131103627 | The C Programming Language | 1978            |
| 978-1491901131 | Learning Python       | 2015            |

**Authors 表：**

| AuthorID | Name       | BirthYear |
| :------- | :--------- | :-------- |
| 101      | Bjarne Stroustrup | 1950      |
| 102      | Brian W. Kernighan | 1942      |
| 103      | Dennis M. Ritchie | 1941      |
| 104      | Mark Lutz  | 1965      |

**BookAuthors 表：**

| ISBN         | AuthorID |
| :----------- | :------- |
| 978-0321765723 | 101      |
| 978-0131103627 | 102      |
| 978-0131103627 | 103      |
| 978-1491901131 | 104      |

**目標：**
請寫出以下 SQL 查詢語句：
1.  查詢所有書名及其出版年份。
2.  查詢出生年份在 1950 年及以後的作者姓名。
3.  查詢書名中包含 "C Programming" 的書籍 ISBN 和書名。
4.  查詢所有由 'Brian W. Kernighan' 撰寫的書籍書名。

**步驟：**
1.  使用 `SELECT` 語句指定要檢索的欄位。
2.  使用 `FROM` 語句指定從哪個表格檢索。
3.  使用 `WHERE` 語句添加過濾條件。
4.  對於涉及多個表格的查詢，使用 `JOIN` 語句來連接表格。

**詳解：**

1.  **查詢所有書名及其出版年份。**
    ```sql
    SELECT Title, PublicationYear
    FROM Books;
    ```

2.  **查詢出生年份在 1950 年及以後的作者姓名。**
    ```sql
    SELECT Name
    FROM Authors
    WHERE BirthYear >= 1950;
    ```

3.  **查詢書名中包含 "C Programming" 的書籍 ISBN 和書名。**
    ```sql
    SELECT ISBN, Title
    FROM Books
    WHERE Title LIKE '%C Programming%';
    ```

4.  **查詢所有由 'Brian W. Kernighan' 撰寫的書籍書名。**
    ```sql
    SELECT B.Title
    FROM Books AS B
    JOIN BookAuthors AS BA ON B.ISBN = BA.ISBN
    JOIN Authors AS A ON BA.AuthorID = A.AuthorID
    WHERE A.Name = 'Brian W. Kernighan';
    ```
    *（這裡使用了表格別名 `AS B`, `AS BA`, `AS A`，這是個好習慣，可以讓查詢更簡潔易讀）*
